BestSeller 当前整本小说构建链路与架构

这份文档只基于当前代码实现，不写未来态。目标是把“现在系统到底怎么写完整一本小说” 讲清楚，包括自动化链路、人工介入点、数据库里的真值对象、质量门禁、导出产物和推荐操作方式。

当前主入口：project autowrite 精控入口：planning import + workflow materialize + project pipeline 真值源：PostgreSQL + pgvector 上下文装配：Path Retrieval → Narrative Tree Search → Hybrid Retrieval

一句话结论

当前系统已经不是“一次大 prompt 写书”，而是一个分层生产流水线： Premise → Planning Artifacts → Story Bible → Chapter/Scene Structure → Narrative Graph → Narrative Tree → Scene Draft → Scene Review/Rewrite → Knowledge Update → Chapter Assemble/Review/Rewrite → Project Review → Repair → Export。

也就是说，现在要完成一本完整小说，最稳的方式不是直接追求“一次成书”，而是先把 规划、约束、上下文、写作、审校、修订、导出 这七层都跑通。

当前推荐工作模式

  1. 先用 小体量试跑 验证题材和模型是否合适。
  2. 再切到 手工精控模式，明确世界观、角色、卷纲、章纲和场景目标。
  3. 让系统跑 scene/chapter/project pipeline。
  4. 看 story-bible / narrative / context / review / project review 的结构化结果。
  5. 对不满意的规划层对象先改，再重跑后续链路。

如果目标是“写一本较强的、可读的、信息密度高的长篇”，不建议直接把 autowrite 当终局；它更适合先出一版可运行草案。

当前完整调用链

1. 项目建立
project create 或 project autowrite
确定 slug、标题、题材、总字数、总章节数。
→
2. 规划生成/导入
planning generate 或 planning import
写入 premise、book_spec、world_spec、cast_spec、volume_plan、chapter_outline_batch。
→
3. Story Bible 物化
workflow materialize-story-bible
把规划 JSON 变成世界规则、地点、阵营、角色、关系、状态快照等真对象。
4. 章节结构物化
workflow materialize-outline
生成 chapter rows 和 scene cards。
→
5. Narrative Graph 物化
workflow materialize-narrative-graph
生成 PlotArc、ArcBeat、Clue、Payoff、EmotionTrack、AntagonistPlan、Contracts。
→
6. Narrative Tree 物化
workflow materialize-narrative-tree
把真值对象导出为目录式叙事树，供路径检索和 tree search 使用。
7. Scene Pipeline
scene draft → scene review → scene rewrite → re-review
通过后刷新场景知识层：Canon、Timeline、CharacterStateSnapshot。
→
8. Chapter Pipeline
assemble → chapter review → chapter rewrite → re-review
必要时先导出当前章草稿给人工看。
→
9. Project Pipeline
project review → repair → export
项目级一致性不过时，会自动尝试一轮 repair。

自动模式 vs 精控模式

模式 入口 适合场景 优点 风险
自动整书模式 project autowrite 快速试题材、试模型、先出草案 最快，链路最短 规划层可控性最低，适合试跑，不适合直接定稿
手工精控模式 project create + planning import/generate + workflow materialize-* + project pipeline 准备认真写一本完整长篇 可以逐层检查世界观、人物、章纲、叙事线 操作更多，但质量和可控性更高
局部重写模式 scene/chapter review|rewrite|pipeline + project repair 修结构问题、修节奏、修一致性 避免整书重跑 前提是规划层已经比较稳定

当前最推荐的整本书操作顺序

  1. 创建项目，先定总字数和目标章节数。
  2. 先产出或导入完整 planning artifacts。
  3. 检查 planning show、story-bible show、narrative show。
  4. 确认 PlotArc、EmotionTrack、AntagonistPlan、Chapter/Scene Contract 没有跑偏。
  5. 再跑 project pipeline 写正文。
  6. 检查 project structure、scene context、chapter context、project review。
  7. 必要时跑 project repair、rewrite cascade。
  8. 最后导出 Markdown / DOCX / EPUB / PDF。

当前不建议的使用方式

  • 直接把一句 premise 丢进去，跑完就当定稿。
  • 在世界观、人物弧光和叙事线还没定下前，就大量重复 repair。
  • 只看导出正文，不看 story-bible / narrative / review 结构化结果。
  • 长篇第一次就上 80 万字或几十章真跑。

数据库与对象层是怎么分工的

层 核心对象 作用
项目元信息 Project、StyleGuide、Volume 定义题材、总字数、章节数、风格和卷级结构。
规划层 PlanningArtifactVersion 存 premise、book/world/cast/volume/outline 等版本化规划产物。
Story Bible WorldRule、Location、Faction、Character、Relationship、CharacterStateSnapshot 提供世界观、角色和状态的可执行真值。
结构层 Chapter、SceneCard 定义本书“写到哪里、每一章每一场要完成什么”。
叙事图谱层 PlotArc、ArcBeat、Clue、Payoff、EmotionTrack、AntagonistPlan、ChapterContract、SceneContract 显式管理主线、副线、感情线、反派推进、伏笔与 contract。
叙事树层 NarrativeTreeNode 把 Story Bible、Arc Graph、Contracts、章节结构导出为路径式目录树，方便确定性检索。
正文层 SceneDraftVersion、ChapterDraftVersion 真正的正文版本库。
知识层 CanonFact、TimelineEvent、CharacterStateSnapshot 把已写内容转成后续可消费的知识和状态。
质量层 ReviewReport、QualityScore、RewriteTask、RewriteImpact 记录审校结论、评分、重写任务和影响面。
工作流层 WorkflowRun、WorkflowStepRun、LLMRun 记录每一步运行、模型调用和产物归属。
导出层 ExportArtifact 存 Markdown / DOCX / EPUB / PDF 的导出记录和路径。

上下文系统现在怎么工作

当前写作不是纯 RAG。优先级是：

  1. Path Retrieval：先按叙事树路径确定当前章/场必读的树节点。
  2. Narrative Tree Search：在相关路径范围内做树搜索。
  3. Hybrid Retrieval：最后才用 pgvector + lexical + structural 做补充检索。

Scene/Chapter context 会显式装配这些内容：

  • Story Bible
  • Recent Scene Summaries
  • Recent Timeline Events
  • Participant Canon Facts
  • Active PlotArcs / ArcBeats
  • Unresolved Clues / Planned Payoffs
  • EmotionTracks / AntagonistPlans
  • ChapterContract / SceneContract
  • Narrative Tree Nodes
  • Retrieval Chunks

为什么这样设计

  • 世界观、人物、叙事线、正文版本、知识层必须分层，不然长篇无法长期维护。
  • 单靠向量检索不稳定，所以先走目录式叙事树，再让向量检索做补充。
  • contract 是“应该写成什么样”，review 是“现在写成了什么样”，两者分开才能做偏差审校。
  • scene 过关后才刷新知识层，避免把错误状态沉淀为 canon。
  • 项目级 review 是最终门禁，避免只看局部章节通过。

长度、章节、场景现在怎么控制

控制层级 现在的控制方式 建议
整本书 project create/autowrite 传入 target_word_count 和 target_chapters 这个决定总体规模，是第一层约束。
默认全局预算 config/default.yaml 中的 words_per_chapter、scenes_per_chapter、words_per_scene 适合调整全局写作风格，不适合精确控章。
章节级 ChapterOutlineInput.target_word_count 认真写长篇时，建议逐章设置。
场景级 SceneOutlineInput.target_word_count 控制节奏最有效，尤其适合爽点场、揭示场、情感场。
修订次数 quality.max_scene_revisions = 2、max_chapter_revisions = 1 避免无限自我修订烧成本。

如果你想认真写一本完整长篇，最该手工控制的是 chapter_outline_batch： 也就是每章目标、每场目标、每场字数、每场入场状态和出场状态。

质量门禁现在是什么

  • Scene Review：看 goal、conflict、emotion、dialogue、style、hook、contract_alignment。
  • Chapter Review：看 goal、coverage、coherence、continuity、style、hook、contract_alignment。
  • Project Review：看 chapter coverage、canon/timeline 覆盖、main plot progression、mystery balance、emotional continuity、character arc progression、world rule consistency、antagonist pressure。
  • Repair：当项目级第一次 review 不通过时，自动 supersede 旧 rewrite tasks，重跑受影响章节，再做一轮项目级 review。

导出逻辑现在是什么

  • scene/chapter/project 任一层没通过 review，并不代表没有产物。
  • 现在即使 requires_human_review=true，系统也会尽量导出当前草稿供人工看。
  • output/<project-slug>/ 下是最终可见产物目录。
  • project autowrite 返回里会给出 output_dir、output_files、export_status。

现在要开始写一本完整小说，最现实的路径

  1. 先选题材和卖点。不要先写正文，先定 premise。
  2. 先跑一个 4 章、1.2 万字的小样，验证模型和题材风格。
  3. 确定题材可行后，进入精控模式，手工改好 book/world/cast/volume/chapter outline。
  4. 把 narrative graph 看一遍，确认主线、副线、感情线、反派线和伏笔线没跑偏。
  5. 再跑全书 pipeline。
  6. 看 project review 和导出正文，不满意的先改 planning 层，不要只靠 repair 死修正文。

建议你下一步怎么开始

  1. 先确定我们要写的书的题材、卖点、读者画像和目标字数。
  2. 先只做一版 Book DNA 和 Book Spec。
  3. 我再帮你把这本书的 World Spec / Cast Spec / Volume Plan / Chapter Outline Batch 拆出来。
  4. 确认结构后，再让系统开始写正文。

一句实话

当前系统已经能完成“整本小说的结构化写作闭环”，但如果目标是“较强可读性、榜单潜力”，关键不在再多跑几次正文， 而在 先把规划层和叙事图谱层定准。这也是为什么现在最推荐的不是直接狂跑 autowrite，而是先明确一本书的世界、人物、线索、卷级任务和章节 contract。
